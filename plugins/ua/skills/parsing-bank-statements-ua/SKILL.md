# parsing-bank-statements-ua

Скіл — як взяти виписку з банку ФОП, обробити і використати для обліку, звірки з інвойсами та декларацією ЄП.

## Ключові джерела

- **ПКУ ст. 292** — касовий метод обліку доходу ФОП на ЄП.
- **ПКУ ст. 164.4, 292.5** — перерахунок валюти за курсом НБУ на дату зарахування.
- **ЗУ «Про платіжні системи та переказ коштів»** (`2346-14`) — формат SWIFT, IBAN.

## Популярні банки для ФОП

| Банк | Формат виписки | API | Зауваги |
|---|---|---|---|
| **mono business** (Universal Bank) | CSV, Excel | Public API з токеном | Найпопулярніший; webhook для транзакцій |
| **Privat24 business** (PrivatBank) | XML, CSV, Excel | API (клієнт-API) | Класика; багато підприємців |
| **Sense Bank** | CSV, Excel | Обмежене API | Сучасний, дешеві SWIFT |
| **FUIB (ПУМБ)** | Excel | Обмежене | Класичний, добрий для Payoneer/Wise |
| **Ощадбанк** | Excel, PDF | Поганий доступ | Раритет для ФОП |
| **Укрсиббанк** | Excel | API | Французький, високі комісії |

## Алгоритм обробки

### Крок 1. Експорт виписки

**mono business:**
1. Застосунок mono business → Операції.
2. Фільтр дата «з-по» (квартал, місяць).

### Крок 3. Класифікація транзакцій

Кожну транзакцію класифікуємо:

| Категорія | Приклад | Дохід ФОП? |
|---|---|---|
| **Оплата від клієнта** | "GOOGLE LLC SWIFT... INV-2026-042" | ✅ Так |
| **Конвертація валюти** | "Конвертація 5000 USD → UAH" | ❌ Внутрішня операція |
| **Комісія банку** | "Комісія за SWIFT" | ❌ Не дохід, не витрата (для ЄП) |
| **Переказ на особисту** | "Переведення на мою картку" | ❌ Внутрішня |
| **Повернення клієнту** | "Повернення за INV-2026-030" | ❌ Зменшення доходу (уточнююча) |
| **Зарплата (вхідна)** | (має бути на особистій, не ФОП) | ❌ Помилковий платіж |

### Крок 4. Підсумок

**Валютний дохід** (після конвертації mono — вже в UAH):

```
Σ_дохід_UAH = Σ(оплати від клієнтів у UAH)
            + Σ(оплати у валюті × курс НБУ на дату зарахування)
            − Σ(повернення клієнтам)
```

**Не включати:**
- Власні перекази.
- Кредит (внутрішній між рахунками).
- Помилкові надходження (повернуті потім).

## Перерахунок валюти — курс НБУ vs банку

### Проблема

Mono business **автоматично конвертує** валютну виручку в UAH за **своїм курсом** (близько НБУ ± 0,2%). На рахунок ФОП приходить уже UAH.

**Питання:** чи декларувати отриману суму UAH (банківський курс) чи перерахувати за курсом НБУ?

### Позиція ДПС (традиційна)

**Дохід ФОП на ЄП = сума зарахувань у UAH**, фактично отримана від клієнта.

Якщо банк конвертує з валюти — беремо суму **в UAH, зараховану на ФОП-рахунок**. Курс НБУ **не застосовується** у цьому випадку (бо валюта не залишилась на рахунку ФОП).

### Позиція (якщо валюта залишається на валютному рахунку)

Якщо клієнт платить в USD і сума залишається на **валютному рахунку ФОП** (без автоматичної конвертації):

**Ст. 292.5 ПКУ:** «Дохід, виражений в іноземній валюті, перераховується у гривні за курсом Національного банку, що діє на дату отримання такого доходу».

→ Курс НБУ **на дату зарахування** на валютний рахунок.

### Практично

**Для mono business з автоконвертацією:**
- Дохід = сума UAH на рахунку ФОП.
- Чек у виписці: «Від GOOGLE LLC... 207 500,00 UAH».

**Для банку без автоконвертації (FUIB, Ощад):**
- Дохід = $5 000 × курс НБУ на дату зарахування.
- Записати обидві цифри в облікову книгу.

## Автоматизація в Python

### Приклад скрипта: mono business CSV → звіт

```python
import pandas as pd

df = pd.read_csv('mono_business_2026_Q2.csv', parse_dates=['Date'])

# Фільтр: тільки вхідні UAH, що не є внутрішніми
df_income = df[
    (df['Amount'] > 0) &
    (df['Currency'] == 'UAH') &
    (~df['Description'].str.contains('Конвертація|Переведення на|Комісія', regex=True, na=False))
]

# Групування за місяцем
monthly = df_income.groupby(df_income['Date'].dt.to_period('M'))['Amount'].agg(['sum', 'count'])
print(monthly)

# Порівняти з декларацією
total_q2 = df_income['Amount'].sum()
print(f"Оборот II квартал: {total_q2:,.2f} грн")
print(f"Очікуваний ЄП 5%: {total_q2 * 0.05:,.2f} грн")
print(f"Очікуваний ВЗ 1%: {total_q2 * 0.01:,.2f} грн")
```

### Матчинг з інвойсами

```python
import re

def extract_invoice_number(description):
    # Шукати INV-YYYY-NNN, АКТ-YYYY-NNN, Акт NNN/MM-YYYY
    patterns = [
        r'INV[-_]?\d{4}[-_]?\d+',
        r'АКТ[-_]?\d+[-_/]?\d*[-_]?\d{4}',
        r'№\s*(\d+)',
    ]
    for p in patterns:
        match = re.search(p, description, re.IGNORECASE)
        if match:
            return match.group(0)
    return None

df_income['Invoice_Match'] = df_income['Description'].apply(extract_invoice_number)

# Join з реєстром інвойсів
invoices = pd.read_csv('invoices_registry.csv')
merged = df_income.merge(invoices, left_on='Invoice_Match', right_on='Number', how='left')

# Неспаровані платежі
unmatched = merged[merged['Number'].isna()]
print(f"Незнайдені інвойси: {len(unmatched)}")
print(unmatched[['Date', 'Amount', 'Description']])
```

## Сценарії

### Сценарій 1: Звірка II кварталу

**Дані:**
- Виписка mono business за квітень-червень 2026.
- Декларація ЄП за II кв. (прогноз: 450 000 грн).

**Алгоритм:**
1. Експортувати CSV за 01.04-30.06.
2. Класифікувати (залишити тільки клієнтські платежі).
3. Підрахувати: чистий оборот UAH = ?
4. Порівняти з 450 000 в декларації.

**Результати:**
- Якщо різниця < 1 грн → збігається.
- Якщо більше → знайти, що не враховано.

### Сценарій 2: Валюта на двох рахунках (UAH + USD)

**Дані:**
- Mono business USD: $10 000 надходжень.
- Потім конверсія $8 000 → UAH на особисту картку.
- Залишок $2 000 на USD-рахунку.

**Дохід ФОП:**
- Отримано на рахунок ФОП (USD): $10 000 × курс НБУ на дату = X грн.
- Ще не конвертоване — **вже дохід** на дату зарахування.
- При конвертації на особисту картку → **не нова операція** (просто переведення).

### Сценарій 3: Клієнт повернув платіж

**Ситуація:**
- Квітень: отримав 50 000 грн від ТОВ «Бета» (інвойс INV-2026-020).
- Травень: через спори — повернув 50 000 грн клієнту.
- **Результат:** немає доходу з цього інвойсу в кварталі.

**Звітність:**
- У квітні — +50 000.
- У травні — −50 000.
- Чистий оборот за квартал — 0 з цього інвойсу.
- Декларація ЄП: реально отриманий чистий оборот.

**Увага:** якщо поверталось **у наступному звітному періоді** → декларація за перший період має повний дохід; повернення зменшує дохід **наступного** періоду. Уточнення через уточнюючу за перший або пряме врахування в поточному — залежно від стану реалізації на момент декларації.

### Сценарій 4: Помилка клієнта (подвійна оплата)

**Ситуація:**
- Клієнт помилково заплатив 50 000 грн двічі.
- Ви повернули зайві 50 000.

**Алгоритм:**
- +50 000 + 50 000 (два зарахування).
- −50 000 (повернення).
- **Чистий дохід = 50 000** (одна операція в декларації).

## Типові помилки

| Помилка | Наслідок | Правильно |
|---|---|---|
| Включити «конвертацію валюти» як дохід | Подвійне декларування | Виключити внутрішні операції |
| Не перевірити вихідні (повернення клієнтам) | Завищення доходу | Чистий оборот = прихід − повернення |
| Використовувати баланс замість оборотів | Неправильна звітність | Обороти (суми зарахувань) |
| Курс НБУ на дату інвойсу замість зарахування | Неправильно | Дата зарахування на рахунок |
| Ігнорувати виписку валютного рахунку | Пропуск доходу | Обидва рахунки (UAH + валюта) |
| Включати перекази з особистої картки | Не дохід | Лише від сторонніх платників |
